## 1. Project Summary

DIN (InfiniteZero) is a federated-learning network coordinated by Solidity contracts on Optimism Sepolia. Raw training data never leaves a participant's machine — only model artifacts (pinned to IPFS) and on-chain coordination state are shared. Participants run `dincli`, a Python/Typer CLI, to act as clients (local training), auditors (scoring), or aggregators (combining models).

The contract layer has two tiers:

- **Platform-level contracts** (deployed once by the DIN-Representative): `DinCoordinator`, `DinToken`, `DinValidatorStake`, `DINModelRegistry` — all under `hardhat/contracts/`.
- **Task-level contracts** (deployed per model by the model owner): `DINTaskCoordinator`, `DINTaskAuditor`.

5. Short write-up: what changed per contract, the proxy pattern chosen and why, and any deviation from the existing `Ownable`/`daoAdmin` access model.

## 5. Key Design Decisions (from PR#13)

**Proxy pattern: Transparent Proxy** over UUPS because upgrade authority is separated from implementation logic — a bad V2 cannot brick the proxy by omitting upgrade functions. Fits the current model where the DIN-Representative controls upgrades; `ProxyAdmin` ownership can transfer to a multisig later.

**Deployment order** (proxies cannot use nested `new Contract()`):
1. Deploy `DinToken` proxy → `initialize()`
2. Deploy `DinCoordinator` proxy → `initialize(dinTokenAddress)`
3. `DinToken.setCoordinator(dinCoordinatorAddress)`
4. Deploy `DinValidatorStake` proxy → `initialize(dinTokenAddress, dinCoordinatorAddress)`
5. `DinCoordinator.updateValidatorStakeContract(stakeAddress)`
6. Deploy `DINModelRegistry` proxy → `initialize(stakeAddress)`

**Access model deviation:** `DINModelRegistry`'s bespoke `daoAdmin` / `setDAOAdmin` / `DAOAdminUpdated` removed in favour of standard OZ `OwnableUpgradeable` (`owner()` / `transferOwnership()`). `dincli` ABIs referencing the old interface will need a follow-up update when testnet is redeployed (out of scope for this PR).

**Storage layout rules:** no reordering/removal of existing variables; new variables only appended; `immutable` cross-contract refs replaced with regular storage; `hardhat-upgrades` `validateUpgrade()` run in tests before each upgrade.
