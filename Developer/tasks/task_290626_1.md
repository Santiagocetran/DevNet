## Objective

Build a `dincli` CLI-level test harness that exercises the full end-to-end lifecycle against a local Hardhat node — from platform contract deployment through a complete Global Iteration. This harness serves two goals:

1. **Regression safety for contract upgrades.** Before any testnet redeployment of upgraded contracts, the harness must pass. It is the blocking prerequisite for WP 6.1.
2. **SDK extraction foundation (P4 WP 1.2).** The test harness exposes which CLI paths are pure orchestration (suitable for the future `dincli/sdk/`) and which are UI/UX wrappers. This separation informs the SDK interface design when `dind` needs to automate the same flows without the Typer layer.

---

## Architecture note — SDK extraction path

The longer-term plan (P4 WP 1.2, September 2026) is to extract `dincli/sdk/` from `dincli/cli/`, so that:

- `dincli/cli/` keeps the Typer commands (UI/UX, argument parsing, Rich output)
- `dincli/sdk/` holds the pure logic (contract calls, IPFS operations, wallet management)
- `dind` (the DIN daemon) calls `sdk/` directly for automated execution and monitoring without going through the CLI layer

The test harness written here should be structured so its helper functions naturally become SDK candidates. Where a test calls a sequence of `dincli` commands, note in a comment which logical operation that sequence represents — that comment will become the SDK function name.

## Implementation notes

- **No containerization in this initial version.** Docker/compose is in scope for WP 0.1 (separate task), not here.
- **Torch environment.** Commands that invoke model training or aggregation must run under `torchenv`. The harness runner should switch environments or document which test groups require which venv.
- **Account index mapping.** Hardhat dev accounts are indexed from 0. Accounts 11–22 are aggregators, 50–58 are auditors, 2–10 are clients. Account 0 is the DIN-Representative; account 1 is the model owner.
- **State assertions.** Each phase should assert the GI state string returned by `task gi show-state 0` matches the expected state from `dincli/cli/utils.py::states`.
- **Idempotency.** Where possible, check whether a contract is already deployed before re-deploying. The harness should be re-runnable on a fresh Hardhat node without manual cleanup. if hardhat node  is already running close it and restart.
