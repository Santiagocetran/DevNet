## Part 1 — P3-T0.2b: Keystore Security & Resource Documentation

### 1a. Audit plaintext key exposure

`dincli/cli/system.py::connect_wallet` already has multiple key-loading paths. The `.env`-based multi-account pattern (`ETH_PRIVATE_KEY_<n>`) is genuinely useful for local testing and multi-account workflows and should be kept. The goal is not to remove these paths but to clearly classify them by security tier and ensure production validators are steered toward the more secure options added in 1c.

Current paths and their appropriate tier:

| Path | Current behaviour | Tier |
|---|---|---|
| `privatekey` positional argument | Raw `0x...` key accepted; warning printed but key enters shell history | Dev only — strengthen the warning |
| `--key-file` / `-f` | Reads plaintext key from a file on disk | Dev/testing — document as not suitable for production |
| `--account <n>` (non-demo) | Reads `ETH_PRIVATE_KEY_<n>` from `.env`; supports multiple accounts | Dev/testing — convenient and useful; not suitable for production validators |
| `_get_password` from `.env` | Encryption password pre-loaded from env var | Dev/testing — acceptable for automation; document the tradeoff |
| Demo mode `WALLET_FILE` | Saves `"private_key"` as plaintext JSON | Dev/Hardhat only — already marked `⚠️ ONLY FOR MOCK!`; no change needed |
| Interactive `getpass` prompt | Prompts for raw key, then encrypts at rest | Acceptable for production key import (key never touches disk) |

The docs and config templates should reflect this classification:

- In `.env.example` files (repo root and `hardhat/`), label `ETH_PRIVATE_KEY_<n>` entries explicitly as development/testing variables — do not remove them, but add a comment directing production validators to the keystore path instead
- In `Documentation/guides/` and `Developer/DEVELOPMENT_SETUP.md`, wherever a guide instructs setting a raw key in `.env`, add a callout: "This pattern is for local development. If you are running a production validator node, use the encrypted keystore — see `Documentation/guides/wallet-setup.md`."
- In `dincli/docker/node/README.md`, add a note in the environment variables section distinguishing dev and production key management

### 1b. Burner wallet setup guide

This is a gap that currently nowhere in the documentation addresses: the existing guides assume the operator already has a wallet and knows why to keep it separate. They do not.

Write `Documentation/guides/wallet-setup.md` with the following sections:

1. **Why a dedicated burner wallet, not your main address.** The private key will be loaded into `dincli` to sign transactions on every command. Any compromise of the machine or the keystore file exposes every asset at that address. Use an address you are comfortable treating as disposable.

2. **Recommended tool — OWS (Open Wallet Standard).** [openwallet.sh](https://openwallet.sh/) is the simplest CLI-based option: chain-agnostic, agent-friendly, generates keystores that follow open standards, and pairs naturally with `dincli`'s workflow. Show the minimal steps to generate a fresh wallet and export its keystore file.

3. **Fund the burner wallet.** Direct the operator to the Optimism Sepolia faucet with the new address. Note the minimum ETH needed to stake and submit a few transactions.

4. **What happens next.** Cross-reference the keystore migration guide (1d) so the operator understands how the key file gets loaded by `dincli` after setup.

This guide should be linked from `Documentation/guides/ipfs.md` and `Developer/DEVELOPMENT_SETUP.md` as a prerequisite step.

### 1c. Multi-account support and security/convenience balance in `connect_wallet`

**Do not remove any existing input paths.** All of them — positional key argument, `--key-file`, `--account <n>` via `.env`, interactive `getpass`, and `_get_password` from `.env` — serve real use cases and must be preserved. The work here is to *add* better options and improve the weakest spots, not to restrict what operators can do.

**The real problem to solve** is that `dincli` currently holds one active account at a time (`WALLET_FILE` is a single file), and the existing paths for multi-account access (`ETH_PRIVATE_KEY_<n>` in `.env`) work well for local testing but have no secure production equivalent. The task is to design and implement multi-account support that works conveniently for everyone — developers and production validators alike — with security that scales to what each user actually needs.

**Known issue: `_cache_password_if_needed`**

This function caches the wallet decryption password as plaintext in a file under `CONFIG_DIR` so users don't have to re-enter it on every command. This is a real security issue (the cached plaintext file is readable by any process with filesystem access) but also a real convenience feature — operators running many sequential commands would otherwise face a password prompt on each one.

The improvement here is not to remove the cache but to **move it from disk to memory**. A short-lived in-memory TTL cache (e.g. a module-level dict with an expiry timestamp, cleared on process exit) gives the same convenience without writing anything to disk. Explore this as the target design: password prompted once per session, held in process memory for a configurable TTL (e.g. 15 minutes), never written to a file.

**What to add:**

- **`--keystore <path>` option** — accept a standard Ethereum JSON keystore file as *input* (already-encrypted, e.g. generated by OWS or `eth-account`). The current code already outputs this format via `Account.encrypt()`; accepting it as input closes the loop for validators who manage their keys externally. Prompt for the decryption passphrase via `getpass`.
- **Named multi-account keystore** — extend `WALLET_FILE` to a named store: multiple encrypted keystores under `CONFIG_DIR` (e.g. `wallet_aggregator.json`, `wallet_auditor.json`), selectable by name via `--account <name>`. This mirrors the convenience of `ETH_PRIVATE_KEY_<n>` for dev, but with encryption at rest for production. A `dincli system list-accounts` command to show what's registered would complete the UX.

**OWS integration exploration:**

Alongside the above, assess whether `dincli` can connect directly to OWS (Open Wallet Standard) as an alternative account backend for operators who want to manage all their keys in one place. OWS is designed as a local agent-accessible wallet layer — if it exposes a programmatic interface (CLI subprocess, local socket, or SDK), `dincli` could query it for a named account and request a signature without ever holding the raw key itself. This is relevant specifically for multi-account production use: an operator with three validator identities could manage all three in OWS and point `dincli` at whichever is needed.

Assess feasibility: what interface does OWS expose, can `dincli` enumerate accounts and request a signed transaction without handling the raw key, and what are the failure modes (OWS not running, account not found, wrong chain)? Write up your findings and a recommendation in the PR description — the outcome shapes whether named multi-keystore or OWS delegation becomes the recommended production path.

**Security/convenience tiers — document in the PR and in `wallet-setup.md`:**

| Path | Convenience | Security | Recommended for |
|---|---|---|---|
| `ETH_PRIVATE_KEY_<n>` in `.env` | High — multi-account, no prompts | Low — plaintext on disk | Local dev, automated testing |
| Interactive `getpass` + encrypted `WALLET_FILE` | Medium — prompt once, in-memory TTL cache | High — key encrypted at rest, password in memory only | Production (current best option) |
| `--keystore <path>` (JSON keystore input) | Medium — passphrase prompt | High — external keystore, key never re-encoded | Production validators with OWS/external key management |
| OWS delegation (if feasible) | High — named accounts, no raw key in `dincli` | Highest — `dincli` never sees the private key | Production validators wanting full key isolation |

**Flag this PR for Umer review before merge.** This is the highest-sensitivity change in the task.

## Part 2 — Filecoin Provider Adapter

**Roadmap ref:** `Developer/discussion/add-filecoin-support.md` — "Now (P3 — can start independently)"

### Context and framing

The document at `Developer/discussion/add-filecoin-support.md` lays out the strategic case for Filecoin-backed storage and evaluates the provider options — Lighthouse, Storacha/Web3.Storage, and Filebase's Filecoin tier — against DIN's architecture. It is a working document written from a protocol design perspective, not a final implementation spec. The implementation decisions — which provider's SDK feels maintainable in practice, where the existing abstraction needs adjusting, what the actual developer experience looks like — are yours to make.

This is an area where DIN is actively evolving, and the choices made here will shape how the network handles storage as it grows. We're bringing you in not just to execute a spec but to apply your judgment: if something in the discussion document doesn't hold up once you're working with the actual APIs, push back and document your reasoning. That kind of grounded input is exactly what helps DIN's technical direction stay practical.

On the blockchain framing: it's largely irrelevant here. The integration point is `dincli/services/ipfs.py` — a pure Python adapter layer with no contract dependencies, no CID format changes, and no Solidity involved. The storage layer happens to be Filecoin-backed rather than Filebase, but the implementation pattern is the same serverless-API-integration work from your BlinkTrip background: a new adapter class behind an existing interface.

The economic layer — per-task storage budgets, sponsored upload credentials, on-chain accounting — is Umer's domain and is being designed in P3-5.3 (fee mechanism, July). What's needed from you now is the provider adapter itself, so operators can begin using Filecoin-backed storage before the full fee model lands.

### What to build

Add a Filecoin-backed provider behind the existing `custom` provider path in `dincli/services/ipfs.py`:

- **Lighthouse** is the recommended starting point — simpler API, fast IPFS retrieval gateway, per-account billing with no raw FIL management. Storacha/Web3.Storage is the stronger long-term fit (native UCAN delegation for sponsored uploads), but Lighthouse is the right first adapter given the scope here. If after reading the discussion document you have a different view, flag it before starting.
- Implement `upload(file_path) -> cid` and `retrieve(cid, dest_path)` using the Lighthouse REST API
- Operator configuration via env vars: `IPFS_PROVIDER=lighthouse`, `LIGHTHOUSE_API_KEY=<key>` — follow the same pattern as the existing `filebase` provider
- Retrieval can fall back to the standard IPFS HTTP gateway; the CID is a CID regardless of which provider stored it
- No changes to any contract or CID handling code

Read `Developer/discussion/add-filecoin-support.md` in full before starting. The "Best Filecoin-backed providers" and "Recommended implementation sequence → Now" sections are the most relevant. If anything in the provider comparison doesn't match what you find when working with the actual SDKs, document the discrepancy — that feedback directly informs the P3-5.3 design work.
